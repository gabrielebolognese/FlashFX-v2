# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

```bash
npm run dev        # Vite dev server (runs predev icon generation first)
npm run build      # Production build (runs prebuild icon generation first)
npm run typecheck  # tsc --noEmit -p tsconfig.app.json (strict; catches unused locals/params)
npm run lint       # ESLint over the repo
npm run icons      # Regenerate public/icons/** chunks from lucide-static
npm run preview    # Serve the production build
npm run verify:cloner  # Acceptance harness for the pure cloner distribution engine
```

There is no test runner configured. Validate changes with `npm run typecheck` and `npm run lint`. For pure, self-contained engines a `scripts/verify-*.mjs` harness may exist that bundles the real TS with the installed esbuild and asserts with `node:assert` (see `scripts/verify-cloner.mjs`) — the convention when a piece is worth proving but a full test runner is still not set up.

`npm run icons` reads `node_modules/lucide-static` and writes the chunked icon JSON under `public/icons/` consumed by the icon library UI. It runs automatically before `dev`/`build`; run it manually after upgrading `lucide-static`. A `preinstall` script writes `ignore-scripts=true` to `.npmrc` to keep dependency postinstall scripts from running.

## Environment & platform requirements

- **WebGPU is required.** The rendering and export pipeline (`WebGPURenderer` in `src/engine/renderer.ts`, `src/codec/exporter.ts`) calls `navigator.gpu.requestAdapter()` with no WebGL fallback. Use a Chromium-based browser with WebGPU enabled.
- **Supabase** connection comes from `VITE_SUPABASE_URL` / `VITE_SUPABASE_ANON_KEY` in `.env`. `src/lib/supabase.ts` exports `supabase` (a client) or `null` when those vars are missing — always null-check it before use so the app degrades gracefully offline.
- Path alias `@/*` → `src/*` (configured in both `vite.config.ts` and `tsconfig.app.json`).

## Architecture

Single-page motion-graphics / short-form video editor ("FlashFX Studio", internal prefix `ffx`). React renders the UI; all animation evaluation, compositing, and video work happen in a WebGPU/worker runtime driving an `OffscreenCanvas`.

### Layered structure

- **`src/core/`** — pure domain model, no React, no rendering. `types.ts` is the source of truth for `Composition`, `Layer`, `Track`, `Keyframe`, `AnimatableProperty`, `Transform`, masks, etc. `factory.ts` constructs layers; `interpolation.ts` evaluates keyframed properties (linear/bezier/hold/spring); `sceneGraph.ts` handles parent/child hierarchy. Feature-specific pure logic lives beside it (`captions.ts`, `silenceDetection.ts`, `textExplode.ts`, `snap/`, `trackCompression.ts`).
- **`src/engine/`** — the runtime. `renderer.ts` (`WebGPURenderer`) rasterizes a resolved frame; `timeline.ts` + `playback.ts` drive the frame clock; `media/`, `video/` (WebCodecs decoder pool, texture cache, frame scheduler), `captions/`, `silence/` each own an isolated subsystem, most backed by **Web Workers**. Caching (`cache/lruCache.ts`, `cache/renderTree.ts`) is central to playback performance.
- **`src/store/`** — Zustand stores that hold editor state and mediate all user mutations. `editor.ts` is the largest and central: it owns the `Composition` and exposes actions. **Undoable mutations go through the command pattern in `store/history.ts`** (`Command` objects pushed to `useHistoryStore`); non-undoable setters are prefixed `_` (e.g. `_setComposition`). When adding an operation that changes the composition, wire it as a history command rather than mutating state directly.
- **`src/ui/`** — React presentation. `ui/layout/` picks a workspace layout (editor vs. builder vs. review vs. short-form); `ui/panels/` are the inspector/timeline/viewport/tool panels; `ui/context-menu/` is a data-driven menu system. Components subscribe to stores with selector functions.

### Feature modules

Many capabilities are self-contained modules under `src/`, each typically exposing a `types.ts`, an `engine.ts`/`store.ts`, and `presets.ts`: `anchoring/`, `physics/` (Rapier 2D via `@dimforge/rapier2d-compat` + `public/rapier_wasm2d_bg.wasm`), `particles/`, `procedural/`, `field-sampling/`, `stagger/`, `expressions/` (worker-evaluated), `layout/`, `animation-items/`, and `animation-builder/` (a node/flowchart authoring mode). Prefer extending the relevant module over threading feature logic through `store/editor.ts`.

Note: consumers import these modules by **deep path**, not through a barrel. The unused `index.ts` barrels were removed in the July 2026 dead-code cleanup; don't reintroduce one unless something actually imports it. `field-sampling/` is **CPU-only** — `fields.ts` (`rasterizeField`/`sampleFieldBilinear`, used by `core/interpolation.ts` and `cloner/`) plus `renderer.ts` (`fieldSampledRenderer`, used by `engine/renderer.ts`). Its WGSL *compute* path (`FieldSamplingEngine`/`fieldWorker`/`shaders/*.wgsl`) was never wired up and has been deleted; recover from git if it's ever wanted.

- **`src/cloner/`** — MoGraph/Cloner (C4D Cloner / AE Repeater): one logical layer that expands into N instances at render time as a pure function of `(params, index)`. **Built incrementally (a multi-prompt feature).** Currently: the schema (plain strict-TS interfaces in `types.ts` — this codebase has no Zod), the **pure distribution engine** `computeInstanceTransforms(cloner, frameNumber, ctx?)` in `distribution.ts` (grid / radial / path modes; `renderCount` is a hard, enforced instance cap), an **effector stack** (`effectors.ts`: `random`/`falloff`/`step`/`time`/`target` → `effectorOutput` + `applyEffectorStack`, ordered, with `add`/`multiply`/`override` blend modes), **timing stagger** (`stagger.ts`: `sourceLocalTime` = `frame − delayForIndex(i)`, reusing the stagger module's `computeStaggerOffsets`), and a reference/cycle validator (`validation.ts`). **Frame-purity is non-negotiable** (the timeline scrubs non-sequentially and must be byte-identical) — the Random effector MUST hash `(seed, index)` with the house `mulberry32`; `Math.random`/`Date` are banned. Reuses `core/motionPath.ts`'s arc-length sampler for the path mode (`pathRef` is a `MotionPath` id — never reimplement arc-length). The source layer's own keyframed animation is evaluated per-instance at its staggered local frame via `ctx.evaluateSourceTransform` — the host injects its **existing** `evaluateNumber`-based evaluator; the cloner never imports `interpolation` (that would drag in the worker-spawning expression engine and break purity). The CPU side of the GPU path: `instanceMatrix.ts` (`composeInstanceMatrix` → column-major 4x4 matching the renderer's **inline** 2D Y-down transform math — the renderer has no existing mat4 convention; `packInstanceBuffer` → interleaved `[mat4(16) | rgb, opacity]` for an instance-step vertex buffer) and `renderPath.ts` (`selectClonerRenderPath` → `instanced-shape` for SDF shapes / `texture-stamp` for image·text·pen-path / reserved `per-instance` for data-bound). **`ClonerLayer` IS now a first-class member of the core `Layer` union** (extended with the common layer fields + optional effect fields), built by `createDefaultCloner`, and **resolved every frame** in `interpolation.ts:resolveFrame` → `ResolvedCloner { renderPath, sourceLayerId, instances }` (source animation reuses the existing `resolveTransform` evaluator at each instance's staggered local frame; `getMotionPath`/`fps` come from the composition). It does NOT draw yet — the renderer's if-chain dispatch skips `layerType:'cloner'` safely. **Field-driven distribution + data binding (Prompt 4):** the `distribution` union has a `field` variant and the falloff-shape union has a `field` variant — both sample a **pre-resolved `FieldGrid`** (the `field-sampling` engine's buffer) via `ctx.getField`, keeping the pure engine synchronous (the async rasterization is resolved once, cached by configJSON, in `resolveClonerField` in `interpolation.ts` — reuses `rasterizeField`/`sampleFieldBilinear`, never a second field impl). `maxCount` (field candidate cap) is distinct from `renderCount` (final cap). `dataBinding.ts:buildInstanceOverrides` generates per-instance content override maps (`data[i % data.length]`, wraparound intentional), and `buildDataBoundSources` APPLIES them to the source via the **instance-override mechanism** in **`src/core/overrides.ts`** (`applyOverrides` — deep-clone + dot-path set; sets an AnimatableProperty's `defaultValue` base rather than replacing it; lenient/no-throw). This is the reusable primitive Prompt 4 called "the precomp instance-override mechanism" (lives in core so a future precomp system shares it). Wired into `resolveFrame`: a data-bound cloner resolves to `ResolvedCloner.instanceSources` — one content-overridden source layer per instance, ready for the (future) full per-instance render path. Routing: data-bound → reserved `per-instance` path; `CLONER_LOD_THRESHOLDS` are per-strategy (per-instance ≪ texture-stamp ≪ instanced-shape). Only the actual RENDER of `instanceSources` remains renderer work (a full precomp-as-composition-reference LAYER type is a separate, unbuilt feature — the override applicator is the piece data binding needed). **Still blocked-by-design for GPU rendering** (unverifiable here): NO existing GPU instancing, NO reusable render-source-to-texture, no exposed scrub signal — the instanced pipeline + texture-stamp RTT + full-render + LOD mechanism are the next chunk. Confirmed decisions: premultiplied source-RTT, SDF-shapes-first, LOD thresholds tuned after profiling. Full GPU design in [`docs/cloner-prompt3-gpu-design.md`](docs/cloner-prompt3-gpu-design.md). Verified by `npm run verify:cloner` (41 checks).

- **`src/core/precomp.ts` + precomp layer type** — Precomposition (a layer that references another `Composition` by id and renders it as a nested `RenderFrame`). **Built incrementally.** Foundation done: `PrecompLayer` in the core `Layer` union (+ `'precomp'` in `TrackType`, `ResolvedLayer.layerType`, and a `ResolvedPrecomp` payload carrying the recursively-resolved nested frame); `createPrecompLayer` factory; `resolveFrame(comp, frame, ctx?)` gained an optional **`ResolveContext {getComposition, depth, visited}`** and a recursive precomp branch that resolves the sub-composition at a time-remapped local frame (`precompLocalFrame` — rebase by inPoint, fps rescale, timeStretch, clamp). **Two hazards were fixed here:** `resolveFrame`'s module-level `_layoutOffsets`/`_layoutContainerSizes` are now **saved/restored around the recursion** (re-entrancy), and a **cycle + `MAX_PRECOMP_DEPTH` guard** (current-comp+ancestors `visited` set) prevents infinite nesting. `validateCompositionGraph` is the **unified** WHITE/GRAY/BLACK cycle detector walking BOTH precomp-nesting AND cloner composition-source edges (`referencedCompositionIds`). **Multi-composition document (Phase B/C, done):** the editor store is now a document — `compositions: Record<id, Composition>` + `rootCompositionId` + `activeCompositionId` + `navStack`, with `composition` kept as the **live active mirror** and `getComposition(id)` merging it (`id===active ? composition : compositions[id]`) so the ~3700-line store's existing `get().composition` reads are untouched. Store actions: `precomposeSelection` (wraps the selection into a sub-comp via the pure `src/core/precompose.ts:buildPrecompose` — preserves world transforms, pulls descendants; one undoable command snapshotting the registry), `enterPrecomp`/`exitPrecomp`/`navigateToComposition` (non-undoable breadcrumb nav, folds the live comp back into the registry on switch), `renameComposition`, `getDocument`/`loadDocument`. **Resolve plumbing:** `TimelineEngine.setResolveContext({getComposition})` (wired from `Viewport.tsx`) threads the registry into `resolveFrame`, so precomps **resolve for real** in preview. **Persistence:** `SceneDocument {version, rootCompositionId, compositions}` in core/types; `serializeDocument`/`deserializeDocument` (project-system) with **migration** of legacy bare-composition scenes; `projects.ts`/`useProjectStore` save/load the document (`saveCurrentProject()` → `getDocument()`; open → `loadDocument`). **`validation.ts` now handles `cloner` + `precomp` layer types** (they were being silently stripped on save/load — this also fixed a latent cloner-persistence bug). Verified by `npm run verify:precomp` (13 checks: recursion, cycle, depth, time-remap, graph validation, precompose, migration, round-trip). **Renderer (Phase D, done — browser-verified only, not runtime-testable here):** `renderFrameUnsafe(frame, target, opts?)` now takes an optional `PrecompRenderOpts {targetView, targetW, targetH, depth, clearAlpha}`; a **pre-pass** recursively renders each precomp layer's nested `RenderFrame` into a **pooled per-precomp texture** (`ensurePrecompTexture`) via its own encoder+submit (so passes never alias the parent's in-flight scene pass), then the layer **composites through the existing image pipeline** — a precomp is bucketed with `imageLayers`, its texture bound as the image's texture with source size = sub-comp w/h, identity filters, and the layer's transform/opacity (straight-alpha, matching how the renderer composites everything else). Bookkeeping (`frameClock`/`renderTree.syncFromLayers`/`activeTextKeys`) is **depth-0-only**, with `pinNestedTextKeys` recursing so a sub-comp's text texture isn't evicted; the render recursion is bounded by the resolve-time depth cap + a `MAX_PRECOMP_DEPTH` backstop. Non-precomp frames render byte-identically (the target is just parameterized). Known v1 limits: differing sub-comp resolution thrashes the shared blur-texture pool per frame; precomp-layer effects (shadow/glow/blur) aren't applied to the composited precomp yet. **UX (Phase E, done):** "Precompose" in the clip context menu (single + multi-select, `menuDefinitions.ts`) and the Toolbar Object menu (`Ctrl+Shift+C` hint — not yet bound to a keydown; the menus/breadcrumb are the wired surfaces); "Open Precomposition" context item; a **breadcrumb bar** (`CompositionBreadcrumb.tsx`, overlaid atop `TimelinePanel`) showing root → precomp → … with clickable segments (`navigateToComposition`) + an up button (`exitPrecomp`), hidden at the root. **The feature is complete end-to-end** (resolve → store → persistence → render → UX); everything except the WebGPU compositing (Phase D) is `tsc`/harness-verified.

### Persistence

- **Local projects** live in **IndexedDB** (`flashfx-projects` DB) via `src/project-system/`. `ProjectApp` is the shell that gates the editor behind a project dashboard; `useAutoSave` persists scenes. Serialization/validation/migration live in `project-system/services/`.
- **Cloud data** (brand kit, saved assets, Google-Drive-backed assets, folders, recovery logs, caption transcripts) is in **Supabase** — see `supabase/migrations/`. Write policies use the **app-key RLS pattern**: the client sends a static `x-app-key` header (set in `src/lib/supabase.ts`) that write policies validate via `app_key_valid()`. This is defense-in-depth, not real auth (no user accounts). The `supabase/functions/drive-assets/` edge function proxies Google Drive.

### Conventions

- TypeScript is **strict** with `noUnusedLocals`/`noUnusedParameters`/`noFallthroughCasesInSwitch` — unused imports/vars fail the typecheck.
- Store actions are the mutation API; components should call actions, not reconstruct composition state inline.
- Keep pure logic in `core/` (and feature modules) unit-testable and free of React/DOM/WebGPU imports so it can run in workers.
- Per `.bolt/prompt`: UI uses **Tailwind** classes and **lucide-react** icons; avoid adding new UI/icon/theme dependencies unless necessary.

### Codebase health (July 2026 cleanup)

`tsconfig.app.json` has `"include": ["src"]`, so **everything under `src/` is typechecked whether or not it ships**. That let an unreachable parallel codebase accumulate: 266 files / ~72k LOC (52%) were dead, generating 73% of the typecheck errors and masking real bugs in live code. All of it was deleted; `npm run typecheck` is the regression gate and should stay **green**.

To keep it that way:

- **A file only ships if something reachable from `src/main.tsx` imports it.** `vite build` proves this — it bundles only reachable modules, so a passing build after deleting a file means that file was dead. Before assuming code runs, check that an import chain reaches it; several documented features turned out to live in orphan files (see the precomp correction above).
- **Ambient `.d.ts` files are the exception** — never imported, but required: `src/vite-env.d.ts` and `src/codec/webcodecs-audio.d.ts` (WebCodecs *audio* types; the TS lib ships only the video ones). Never delete these by import-graph reasoning.
- **Don't silence typecheck errors** with `any`/`@ts-ignore`/`!` or by relaxing tsconfig. The errors that survived the cleanup were real defects (wrong arg counts, properties that don't exist) — the noise was hiding them.
- Removed dead deps: `framer-motion`, `three`, `@types/three` (nothing, including Spline, depends on `three`).
